Background Subtraction

A video is an “image stack”, meaning it is composed of a stack of images, each one called a frame, each taken at a specific time.

We can thus look at video data as a spatio-temporal volume. If the camera is stationary, each line through time corresponds to a single ray in space — the value of the pixel in a specific x and y position in the video across time can be recorded as it changes.

Our goal in this part is to find a model for the background. This can be useful in many applications such as in surveillance. We can use the background model to find out which elements in the foreground have changed and to identify motion in the scene.

Suppose we have an input image that looks like the following. It’s a video of the bottom tiles of a pool, with water ripples on the surface. If we take 10 frames of the video and average them, we cancel out the wave effect. The resultant image is blurry but the background is clear. We have simply obtained a background model. We can use this background model to identify what is in the foreground — namely in this example, the water ripples.

Another reason why this is useful, is instead of processing the entire image, background and all, we can extract the foreground — which is what we are interested in anyway — and ignore everything that goes on in the background. This saves a considerable amount of computation power and time.

Background subtraction is a simple technique in theory that does well with a static camera, but is imperfect in many ways. For one thing, post processing is necessary to find which points in the foreground belong to the same object (usually modelled as: if pixels move together, then they belong to the same object, again simple in theory, not so simple to implement).

Background subtraction has many applications:

  • Traffic monitoring (counting vehicles, detecting & tracking vehicles, pedestrians),
  • Human action recognition (run, walk, jump, squat),
  • Human-computer interaction,
  • Object tracking

2 Mean Filter to find the background model

Note how the car closest to the camera is still present as a ghost image in the background image and in the foreground is detected incorrectly. This is because cars moving slower or closer to the camera will appear in a lot of frames and don’t get removed completely. A way to fix this is to increase n, the number of frames we average on. One downside to this approach in comparison the mean, is that you need to store all the frames in memory to compute the median, whereas with the mean you can use a moving average to avoid having to use up the entire memory.

Pros and cons of the previous approaches

Advantages:

  • Extremely easy to implement and use!
  • All pretty fast.
  • Corresponding background models need not be constant, they change over time.

Disadvantages:

  • Accuracy of frame differencing depends on object speed and frame rate
  • Median background model: relatively high memory requirements
  • Setting global threshold Th

What happens if the background is not so simple?

Background mixture models

Below we have 3 complex examples of backgrounds.

The first image has a background that contains a large shadow. The shadow’s position may change over time and it needs to still be included in the background model. In the second image, there is a sunset, so the entire image’s values changes over time and so this also needs to be coded in. The third image is taken from a video where the open door gets closed. This changes the background entirely, and our model needs to be able to handle this.

Idea: model each background pixel with a mixture of Gaussians; update its parameters over time. The final model is the envelope of all these gaussians combined.

Action Recognition (Read only, not included in the final)

What is an action?

An action is a transition from one state to another. In computer vision, we try to extract useful information from a video such as:

  • Who is the actor?
  • How is the state of the actor changing?
  • What (if anything) is being acted on?
  • How is that thing changing?
  • What is the purpose of the action (if any)

We can represent actions by relating each action to a

  • pose (defined as the configuration of body parts),
  • a category
    • (examples: walking, hammering, dancing, skiing, sitting down, standing up, jumping),
  • or in the form of nouns and predicates
    • <man, swings, hammer>
    • <man, hits, nail, w/ hammer>

In short, we wish to encode temporal relationships to describe action. How do we encode action? We can either use appearance or motion. Appearance can include the

  • pose of the entity in the image,
  • the object held, or
  • nearby objects or scenery.

What about motion? We can encode motion in several ways:

  • using optical flow channels: we can add an extra channel aside from our usual RGB channels, that describe the optical flow. However, this gives poor performance for real-time applications.
  • computing the difference between 2 frames (frame differencing): this is much more suitable for real-time.

Motion History

Motion history is one way to represent motion. The only requirement is that we have a controlled environment (usually boils down to having a stationary camera or at least knowing the motion of the camera).

Below is an example for an optical flow channel that retains its motion history. Space-Time volumes

Another method is to do the same thing as with motion history but instead to accumulate over space, x and y, rather than just time. (This was not discussed at great length in the lecture)

Optical Flow with Split Channels

The optical flow channels can be split further into positive and negative channels, then blurred. (This was not discussed at great length in the lecture)

Pixel Trajectories

Tracking pixels and finding out their trajectories can also give us an indication for the motion that has occurred in the video. One can find the rate of change in the x and y directions through time, and use this to describe the motion.

Interest Points

We can generalize Harris Corner detection and blob detection algorithms (lecture 4) so that they work through time. We then have a moving corner or interest point and we track how it changes through time.

Action Recognition using Feature-based Classification

Recall the usual pipeline for image classification

Also recall spatial pyramids we used in lecture 4, where we divided each image into a set of pixels and computed histograms for these sets alone.

We can utilize these in videos for action recognition using several methods:

1. Spatio-temporal pyramids

In this method, image gradients and optical flows are computed, the histograms are computed for each block, then these blocks are concatenated together to form the feature descriptor (this was not discussed at length in the lecture).

2. Spatio-temporal interest points

Use the interest points across time as the features.

Searching the video for an action

This involves 2 steps:

  1. Detect keyframes using a trained HOG detector in each frame
  1. Classify the detected keyframes either as positive (e.g. “drinking”) or negative (”other”)

Using keyframe detection considerably increases precision in results.

The components we need for action classification are:

  • Space-time interest point detectors (STIP)
  • Descriptors
    • HOG (Histogram of Oriented Gradients)
    • HOF (Histogram of Oriented Flows)
  • Pyramids histograms
  • SVMs with Chi-Squared kernel

Conv-Nets

Conv-nets originally deal only with spatial streams, so to use them with videos, 2 streams are required: the spatial stream and the temporal stream

Note that unlike in image processing, simply using conv-nets did not give better results over traditional results in dealing with images. More processing and techniques had to be tailored to achieve better results.

Alignment and tracking

In the following series of frames, there were two people each walking towards the other’s direction. The detection boxes were able to show that there are 2 people in the image up to a point when they align and it can no longer differentiate between the 2. When they separate, the detection boxes return. This is the problem we are trying to solve in this section.

The image alignment problem is that we have a template and we wish to find it in the video.

On the frame level we have multiple ideas:

  • Template matching → a slow solution where we pass the template over the entire image in order to find a patch that matches it.
  • Pyramid template matching → if the we are not sure what size the template is
  • Model refinement → if we have an initial guess for the position of the template in the image, then we can use this. For multiple frame, if we assume that the difference between the two frames is not that big, then this solution is suitable, as we assume we know where the template was in the previous frame and can thus search around the area. This is a much faster solution than the previous ideas.

Kanade-Lucas-Tomasi (KLT) tracker

Now that we have solved the problem known as image alignment, how can we use this to solve the tracking problem? The two problems are closely related, since in tracking we assume that the template is the object we wish to track and we wish to do successive alignment through the video, assuming we have a starting bounding box on that object.

The KLT tracker chooses good features to track, namely the corners of the object.

How do we choose those good features to track?

Recall moments ago when we derived the Lucas-Kanade image alignment algorithm. A key requirement for the algorithm was that the Hessian matrix, HHH﻿ be invertible.

Inversion fails when the matrix HHH﻿ is singular.

A well-conditioned matrix is defined when both eigenvalues have a similar magnitude and both are large.

As an example we can consider the translation model (W\bold W W﻿ is 2x2 in this case not 2x6!)

Notes on the motion module

Throughout the previous two lectures, we found stark differences in dealing with videos in comparison to images. Videos do not need just the RGB channels in order to find good features; further processing is required and channels must be added before even the almighty conv-nets can try to make a dent in the accuracies of the results. So, in short, videos are a handful, and not just in the fact that they need better computational powers and larger memories to process them.
